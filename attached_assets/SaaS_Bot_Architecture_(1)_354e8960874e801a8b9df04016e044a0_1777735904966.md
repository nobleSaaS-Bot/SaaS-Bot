# SaaS Bot Architecture (1)

This page documents the full architecture of the Telegram SaaS Platform — how the codebase is structured, what each layer does, and how everything connects.

---

## 🗺️ High-Level Architecture Overview

The platform is a multi-tenant Telegram SaaS that lets businesses run full e-commerce stores inside Telegram bots. Here's how the major layers interact:

```mermaid
flowchart TD
    A["👤 Customer / Telegram User"] -->|"Sends message"| B["🤖 Telegram Bot Engine\n(bot_service, handlers, checkout)"]
    B -->|"Webhook"| C["🌐 FastAPI Backend\n(main.py + routes/)"]
    C -->|"Auth & Security"| D["⚙️ Core Systems\n(billing, limits, plans, security)"]
    C -->|"Business logic"| E["🧠 Services Layer"]
    E --> F["🛍️ Storefront Service"]
    E --> G["💳 Payment Providers\n(Stripe, Telebirr, M-Pesa)"]
    E --> H["🤖 AI Layer\n(store builder, product gen, branding)"]
    E --> I["📦 Order Service"]
    E --> J["📊 Analytics Service"]
    E --> K["⚙️ Flow Engine"]
    C -->|"Reads/Writes"| L["🗄️ Database\n(PostgreSQL via SQLAlchemy)"]
    C -->|"Enqueues jobs"| M["⚡ Redis / RQ Workers\n(flow, payment, subscription jobs)"]
    N["🖥️ React Frontend\n(Dashboard, AI Builder, Orders…)"] -->|"REST API"| C
    O["📡 External Webhooks\n(Stripe, Telebirr, M-Pesa, Telegram)"] --> C
```

### Layer Summary

| Layer | What it does | Key files |
| --- | --- | --- |
| **Telegram Engine** | Handles all bot interactions — messages, checkout UI, session state | `services/telegram/` |
| **FastAPI Backend** | REST API, webhook receivers, routing | `app/main.py`, `routes/` |
| **Core Systems** | Subscription enforcement, plan limits, JWT auth | `core/` |
| **AI Layer** | AI-powered store creation, product generation, branding | `services/ai/` |
| **Flow Engine** | Custom bot conversation flows defined by merchants | `flow_engine.py`, `flow_executor.py` |
| **Payments** | Multi-provider payments (Stripe, Telebirr, M-Pesa) | `services/payments/` |
| **Background Jobs** | Async processing for flows, payments, subscription renewals | `workers/` |
| **Frontend** | React merchant dashboard for managing store, products, orders | `apps/frontend/` |

## 🔄 Data Flow

This section describes how data moves through the system for the three core scenarios.

### 1. Customer Places an Order via Telegram

```mermaid
sequenceDiagram
    participant U as Customer
    participant TG as Telegram
    participant BE as FastAPI Backend
    participant FE as Flow Engine
    participant PS as Payment Service
    participant DB as Database
    participant WK as RQ Worker

    U->>TG: Sends message to bot
    TG->>BE: Webhook (telegram.py)
    BE->>FE: Execute conversation flow
    FE->>BE: Return bot response
    BE->>TG: Send reply to customer
    U->>TG: Confirms order
    TG->>BE: Webhook (checkout)
    BE->>PS: Initiate payment
    PS->>DB: Save pending order
    PS-->>U: Payment prompt
    U->>PS: Completes payment
    PS->>BE: Payment webhook (Stripe/Telebirr/Mpesa)
    BE->>WK: Enqueue payment_job
    WK->>DB: Update order status
    WK->>TG: Notify customer
```

### 2. Merchant Builds a Store with AI

```mermaid
sequenceDiagram
    participant M as Merchant
    participant FE as React Frontend
    participant BE as FastAPI Backend
    participant AI as AI Layer
    participant DB as Database
    participant SS as Storefront Service

    M->>FE: Opens AI Builder
    FE->>BE: POST /ai/build-store
    BE->>AI: ai_store_builder.py
    AI->>AI: ai_product_generator + ai_branding
    AI->>AI: ai_pipeline.py orchestrates
    AI->>DB: Save store, products, categories
    AI->>SS: Register subdomain
    BE-->>FE: Store URL + preview
    FE-->>M: Show live store preview
```

### 3. Subscription Lifecycle

```mermaid
sequenceDiagram
    participant M as Merchant
    participant BE as FastAPI Backend
    participant CS as Core Systems
    participant DB as Database
    participant WK as RQ Worker

    M->>BE: Subscribes to a plan
    BE->>CS: billing.py enforces plan
    CS->>DB: Save subscription record
    WK->>DB: subscription_jobs checks renewals
    WK->>BE: Trigger renewal charge
    BE->>CS: limits.py enforces feature caps
    CS-->>M: Access granted or restricted
```
